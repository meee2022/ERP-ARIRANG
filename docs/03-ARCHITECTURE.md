# ERP V1 вЂ” Final Technical Architecture

## DECISIONS (final, no alternatives)

---

## Frontend Stack
- **Framework**: Next.js 16 (App Router) вЂ” already installed
- **Language**: TypeScript 5 вЂ” already installed
- **Styling**: Tailwind CSS v4 вЂ” already installed
- **Component Library**: shadcn/ui (Radix UI primitives) вЂ” already installed
- **State Management**: Zustand вЂ” already installed (useAppStore)
- **Charts**: Recharts вЂ” already installed
- **Icons**: lucide-react вЂ” already installed
- **Date handling**: date-fns вЂ” already installed
- **RTL**: dir="rtl" on html element. All layout uses logical CSS properties (start/end not left/right)
- **Fonts**: Tajawal (Arabic, Google Fonts) as primary. Inter as fallback for English
- **i18n**: Custom implementation using a simple dictionary object in lib/i18n.ts + useLocale() hook. No external i18n library needed for V1.

## Backend Stack
- **Backend**: Convex вЂ” already installed and configured
- **Convex version**: 1.35.1 (current)
- **Convex functions**: mutations (write), queries (read), actions (for external calls if needed)
- **Convex transactions**: All posting operations in a single mutation = atomic by Convex design

## Database
- **Database**: Convex (document store with server-side ACID transactions per mutation)
- **Schema**: Defined in convex/schema.ts using defineTable/defineSchema
- **Indexing**: .index() on all FK fields and commonly filtered fields
- **Numeric precision**: Numbers stored as JavaScript numbers (IEEE 754 double). All monetary values kept to max 2 decimal places enforced in application layer. No integer multiplication strategy вЂ” Convex handles numbers natively.

## Auth
- **V1**: Simple role-based auth using Zustand store (in-memory after login)
- **Session**: Stored in localStorage + Zustand
- **Passwords**: bcrypt on server side (Convex action)
- **V2**: Upgrade to Convex Auth with proper JWT tokens

## File Storage
- **Attachments**: Convex file storage (built-in)
- **Export files**: Generated client-side (CSV) or via Convex action (PDF using jsPDF)
- **Imports**: CSV/Excel parsed client-side using xlsx library (already installed)

## Reporting
- **All financial reports**: Computed via Convex queries from journal_lines table
- **Heavy reports (Trial Balance)**: Convex paginated query with aggregation
- **Export**: PDF via jsPDF (install), Excel via xlsx (already installed)
- **Print**: Browser print API with print-specific CSS

## Background Jobs
- **Convex Scheduled Functions**: Used for period-end reminders, due cheque alerts
- **V1 scope**: Minimal вЂ” just due cheque alert scheduling

## Search
- **V1**: Convex .searchIndex() on accounts, items, customers, suppliers by name and code
- **No external search engine in V1**

## Multi-Branch Model
- **Architecture**: Single Convex deployment, all data has companyId + branchId
- **Branch isolation**: Every query filters by user's authorized branchIds (from Zustand store)
- **Consolidated view**: Available to admin/manager roles вЂ” queries without branch filter
- **Branch selector**: In app header, stored in Zustand, applied to all data queries

## Audit Logging
- **Strategy**: Every Convex mutation that modifies data calls a shared `logAudit()` helper
- **Storage**: auditLogs table in Convex (append-only вЂ” no mutation to update/delete audit records)
- **Content**: entity type, entity ID, action, old values (JSON), new values (JSON), userId, timestamp
- **Retention**: All audit logs kept permanently in V1

## Deployment Architecture
- **Frontend**: Vercel (Next.js deployment)
- **Backend**: Convex Cloud (managed)
- **Environment**: .env.local with NEXT_PUBLIC_CONVEX_URL
- **Domains**: Custom domain on Vercel in production

All Convex functions follow this pattern:

```typescript
// List with filters

  args: {
    branchId: v.optional(v.id("branches")),
    from: v.optional(v.string()),
    to: v.optional(v.string()),
    status: v.optional(v.string()),
    customerId: v.optional(v.id("customers")),
  },
  handler: async (ctx, args) => { ... }
});

// Single document with related data
export const getSalesInvoice = query({
  args: { invoiceId: v.id("salesInvoices") },
  handler: async (ctx, args) => { 
    // returns invoice + lines + journal entry + customer
  }
});
```

### Mutations (write operations)
```typescript
export const postSalesInvoice = mutation({
  args: { invoiceId: v.id("salesInvoices"), userId: v.id("users") },
  handler: async (ctx, args) => {
    // 1. Fetch invoice + lines
    // 2. validatePeriodOpen()
    // 3. Check document_status = approved
    // 4. Build journal entry using buildSalesInvoiceJournal()
    // 5. postJournalEntry() вЂ” atomic
    // 6. updateStockBalance() per line вЂ” atomic
    // 7. Update invoice: postingStatus=posted, journalEntryId, postedBy, postedAt
    // 8. Update paymentStatus based on invoice type
    // 9. logAudit()
    // All inside single mutation = atomic
  }
});
```
